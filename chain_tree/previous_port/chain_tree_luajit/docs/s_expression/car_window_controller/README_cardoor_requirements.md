# Car Door Window Controller — Behavior Tree Requirements

## Overview

This document defines the behavior tree architecture for a car power window controller built around the Infineon TLE7269G H-bridge motor driver IC. The tree specifies four concurrent subsystems — serial command handling, motor state control, status reporting, and continuous diagnostics monitoring — that coordinate through a shared blackboard and event system.

## Origin

The behavior tree specification was generated by an LLM (Claude, Anthropic) reading the TLE7269G datasheet and supporting documentation. The LLM extracted the chip's operating modes (forward, reverse, brake, freewheel), fault detection capabilities (overcurrent, thermal shutdown, fault pin), and diagnostic registers, then synthesized them into a complete control architecture expressed as a hierarchical behavior tree.

The resulting specification captures the essential control logic that the TLE7269G requires — bridge mode sequencing, current monitoring, fault response, and the brake-before-freewheel shutdown pattern — but should be reviewed by the hardware engineer for application-specific thresholds, timing, and safety requirements.

## Architecture

The controller is structured as four concurrent branches under a single root node:

**Branch 1 — Serial Message Handler.** Waits for incoming serial data, parses the command type, and dispatches to the appropriate event (open, close, stop, status, emergency). This branch operates as a pipeline: wait for data, read/parse, then dispatch.

**Branch 2 — Motor State Machine.** The core control logic with five states: idle, opening, closing, auto-reverse, and emergency. Each motion state (opening/closing) runs three concurrent monitors — current sensing, position limit switches, and obstacle detection — while driving the motor. The auto-reverse state provides a brief reverse pulse after an obstruction during closing, which is a common anti-pinch safety feature in automotive window controllers. The emergency state kills the motor immediately and waits for a manual reset before returning to idle.

**Branch 3 — Status Reporter.** Responds to explicit status requests and periodic 1-second triggers by reading diagnostics from the TLE7269G, current position, motor current, and temperature, then assembling and transmitting a status message over serial.

**Branch 4 — Diagnostics Monitor.** Three continuous monitoring loops running in parallel: thermal protection (chip temperature vs. threshold), supply voltage monitoring (under/over voltage), and TLE7269G fault pin detection. Any fault in this branch posts an emergency stop event that preempts the motor state machine.

## Interrupt-Level vs. Tree-Level Partitioning

In a real implementation, certain functions would be handled at the interrupt level rather than within the behavior tree tick loop:

**Interrupt-level (hardware ISR):**
- Serial byte reception (UART RX interrupt fills a buffer; the tree checks a "data available" flag)
- Overcurrent detection via the TLE7269G's hardware current sense — a comparator interrupt can kill the motor within microseconds, far faster than any software tick
- TLE7269G fault pin assertion — a GPIO interrupt on the fault output provides immediate hardware-level response
- Emergency stop from a dedicated hardware input (e.g., pinch-detect sensor) — the ISR disables the H-bridge outputs directly, then sets a flag for the tree to handle the recovery sequence
- ADC conversion complete interrupts for current and temperature sampling
- Timer interrupts for PWM generation and tick timing

**Tree-level (behavior tree tick):**
- All state machine logic and state transitions
- Command parsing and dispatch
- Motor startup/shutdown sequencing (bridge mode transitions, brake hold timing)
- Concurrent monitor evaluation (checking flags set by ISRs)
- Diagnostic data collection and status reporting
- Auto-reverse sequencing
- Fault flag management and manual reset handling

The key insight is that the tree structure remains intact — the ISRs simply set blackboard flags and post events that the tree reads on each tick. The monitors in the tree (current sense, position sense, obstacle detection) check these ISR-updated flags rather than polling hardware directly. This gives you microsecond-level hardware protection combined with deterministic, testable behavior tree logic for all the sequencing and coordination.

## Implementation Notes

**TLE7269G Bridge Modes.** The specification uses four bridge states: freewheel (coasting), forward (open direction), reverse (close direction), and brake (active braking with both low-side FETs on). The shutdown sequence always applies brake before transitioning to freewheel — this prevents the window from coasting under its own weight.

**Anti-Pinch (Auto-Reverse).** During closing, an obstruction triggers a brief reverse pulse at half speed before returning to idle. This is a regulatory requirement in many automotive markets (EU, US FMVSS 118). The behavior tree handles this as a distinct state to keep the sequencing clear.

**Three Concurrent Monitors.** The opening and closing states each run current monitoring, limit switch detection, and stall detection in parallel. Any of these can terminate the motion by setting a blackboard flag. The main sequence waits on an OR predicate across all five stop conditions (position reached, stop request, overcurrent, obstruction, emergency).

**Fault Escalation.** All diagnostic faults (thermal, voltage, TLE7269G fault pin) escalate to the emergency state via an event. The emergency state is the highest priority in the motor state machine and preempts any active motion.

## Resource Estimate

When compiled to the S-Expression Engine DSL (v5.3, 32-bit target):

| Resource | Estimate |
|---|---|
| Behavior tree nodes | ~200–280 |
| Node state RAM | ~800–1120 bytes |
| pt_main pointer slots | 4 (32 bytes) |
| Blackboard (22 int32 + 7 float + 2 ptr64) | 132 bytes |
| **Total runtime RAM** | **~1.0–1.3 KB** |

The compiled binary tree is stored in ROM (zero-copy loading) and does not consume RAM. This fits comfortably on a 32KB ARM Cortex-M0/M0+ target with room for the application stack, ISR handlers, and peripheral drivers.
