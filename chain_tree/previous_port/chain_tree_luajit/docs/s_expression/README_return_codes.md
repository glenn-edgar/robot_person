# S_Engine Return Codes

## Overview

The S_Engine uses a layered return code system that maps to three distinct scopes of control flow. Each layer serves a different purpose in the execution hierarchy, from ChainTree event propagation down to internal pipeline sequencing.

## The Three Layers
```
┌─────────────────────────────────────────────────────────────────┐
│                      ChainTree Walker                           │
│                   (behavior tree engine)                        │
└──────────────────────────┬──────────────────────────────────────┘
                           │
                           │ APPLICATION RESULT CODES
                           │ (SE_CONTINUE, SE_HALT, SE_TERMINATE, ...)
                           │
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│                     S_Engine Function                           │
│              (virtual function / subtree)                       │
└──────────────────────────┬──────────────────────────────────────┘
                           │
                           │ FUNCTION RESULT CODES
                           │ (SE_FUNCTION_HALT, SE_FUNCTION_RESET, ...)
                           │
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│                    Pipeline Composites                          │
│         (sequence, state_machine, fork, parallel)               │
└─────────────────────────────────────────────────────────────────┘
                           │
                           │ PIPELINE RESULT CODES
                           │ (SE_PIPELINE_TERMINATE, SE_PIPELINE_RESET_CONTINUE, ...)
                           │
                           ▼
                    ┌──────────────┐
                    │ Child Nodes  │
                    └──────────────┘
```

## Application Result Codes (0-5)

These codes pass through the S_Engine unchanged. They originate from ChainTree's event system and control tree-level propagation. When a node returns one of these codes, the S_Engine forwards it directly to the ChainTree walker.
```c
SE_CONTINUE           = 0,  // Normal execution, continue to next tick
SE_HALT               = 1,  // Pause execution, retain state
SE_TERMINATE          = 2,  // Complete execution, trigger termination
SE_RESET              = 3,  // Restart from initial state
SE_DISABLE            = 4,  // Deactivate node, skip future ticks
SE_SKIP_CONTINUE      = 5,  // Skip remaining siblings, parent continues
```

### Semantics

| Code | Tree Effect | State Effect |
|------|-------------|--------------|
| `SE_CONTINUE` | Walker proceeds normally | Node remains active |
| `SE_HALT` | Walker pauses this branch | Node retains current state |
| `SE_TERMINATE` | Walker terminates this subtree | Termination handlers run |
| `SE_RESET` | Walker resets this subtree | All nodes re-initialize |
| `SE_DISABLE` | Walker skips this node | Node marked inactive |
| `SE_SKIP_CONTINUE` | Walker skips siblings | Parent composite continues |

### Pass-Through Behavior

The S_Engine tick functions pass these codes through transparently:
```c
s_expr_result_t s_engine_tick(s_engine_t* engine, cfl_event_t event) {
    s_expr_result_t result = execute_root_node(engine);
    
    // Application codes pass through to ChainTree
    if (result <= SE_SKIP_CONTINUE) {
        return result;  // ChainTree handles propagation
    }
    
    // Function and pipeline codes handled internally
    // ...
}
```

This ensures the S_Engine is a transparent layer — ChainTree events flow through without the S_Engine needing to understand their full semantics.

## Function Result Codes (6-8)

These codes affect the **entire S-expression function** currently executing. They are generated by nodes within the S-expression and processed by the S_Engine's function-level handler.
```c
SE_FUNCTION_HALT      = 6,  // Halt the entire S-expression function
SE_FUNCTION_RESET     = 7,  // Reset the entire S-expression function
SE_FUNCTION_TERMINATE = 8,  // Terminate the entire S-expression function
```

### Use Case: Exception-Like Control Flow

Function-level codes provide a way to escape from deeply nested structures without unwinding through each layer manually.
```lisp
;; Deep nesting - how does inner node abort the whole function?
(pipeline
  (sequence
    (fork
      (if_then_else
        (critical_error_detected)
        (return SE_FUNCTION_TERMINATE)  ;; Escapes everything
        (normal_operation)))))
```

Without function-level codes, the `critical_error_detected` path would need to return a code that each parent composite understands and propagates. With `SE_FUNCTION_TERMINATE`, the S_Engine catches it at the function boundary.

### Processing
```c
s_expr_result_t execute_s_function(s_engine_t* engine) {
    s_expr_result_t result = execute_root_composite(engine);
    
    switch (result) {
        case SE_FUNCTION_HALT:
            // Halt this function, return SE_HALT to ChainTree
            return SE_HALT;
            
        case SE_FUNCTION_RESET:
            // Reset all nodes in this function
            reset_all_node_states(engine);
            return SE_CONTINUE;  // Or SE_RESET depending on semantics
            
        case SE_FUNCTION_TERMINATE:
            // Clean termination of this function
            run_termination_handlers(engine);
            return SE_TERMINATE;
            
        default:
            // Application codes pass through
            return result;
    }
}
```

### Mapping to Application Codes

Function codes translate to application codes at the function boundary:

| Function Code | Translated To | Rationale |
|---------------|---------------|-----------|
| `SE_FUNCTION_HALT` | `SE_HALT` | Function paused, ChainTree sees halt |
| `SE_FUNCTION_RESET` | `SE_CONTINUE` | Function restarted internally, ChainTree continues |
| `SE_FUNCTION_TERMINATE` | `SE_TERMINATE` | Function complete, ChainTree terminates |

## Pipeline Result Codes (9-12)

These codes are internal to **composite nodes** that sequence children: `pipeline`, `sequence`, `state_machine`, `fork`, `parallel`, etc. They control how the composite manages its child execution.
```c
SE_PIPELINE_TERMINATE      = 9,   // Terminate this pipeline, continue parent
SE_PIPELINE_RESET_CONTINUE = 10,  // Reset pipeline children, continue execution
SE_PIPELINE_RESET_HALT     = 11,  // Reset pipeline children, halt execution
SE_PIPELINE_DISABLE        = 12,  // Disable this pipeline
```

### Scope: Composite-Local

Pipeline codes do not propagate beyond their immediate composite parent. A `sequence` node processes these codes internally:
```c
s_expr_result_t sequence_tick(sequence_t* seq) {
    for (int i = seq->current_child; i < seq->child_count; i++) {
        s_expr_result_t result = invoke_child(seq, i);
        
        switch (result) {
            case SE_CONTINUE:
                // Move to next child
                continue;
                
            case SE_PIPELINE_TERMINATE:
                // End this sequence successfully
                seq->current_child = 0;
                return SE_CONTINUE;  // Parent sees normal completion
                
            case SE_PIPELINE_RESET_CONTINUE:
                // Reset all children, restart from beginning
                reset_children(seq);
                seq->current_child = 0;
                continue;  // Keep executing
                
            case SE_PIPELINE_RESET_HALT:
                // Reset all children, pause
                reset_children(seq);
                seq->current_child = 0;
                return SE_HALT;  // Parent sees halt
                
            case SE_PIPELINE_DISABLE:
                // Disable this sequence
                seq->enabled = false;
                return SE_DISABLE;
                
            default:
                // Application or function codes bubble up
                return result;
        }
    }
    return SE_CONTINUE;
}
```

### Use Cases

**SE_PIPELINE_TERMINATE**: Early exit from a sequence when goal is achieved.
```lisp
(sequence
  (try_method_a)
  (if (success) (return SE_PIPELINE_TERMINATE))
  (try_method_b)
  (if (success) (return SE_PIPELINE_TERMINATE))
  (try_method_c))
```

**SE_PIPELINE_RESET_CONTINUE**: Retry loop pattern.
```lisp
(sequence
  (attempt_connection)
  (if (connection_failed)
    (begin
      (log "Retrying...")
      (delay 1000)
      (return SE_PIPELINE_RESET_CONTINUE)))
  (send_data))
```

**SE_PIPELINE_RESET_HALT**: Reset and wait for external event.
```lisp
(state_machine current_state
  ;; State 0: Idle
  (sequence
    (wait_for_trigger)
    (set_state 1))
  
  ;; State 1: Running
  (sequence
    (perform_action)
    (if (abort_requested)
      (begin
        (cleanup)
        (set_state 0)
        (return SE_PIPELINE_RESET_HALT)))  ;; Reset state 1, halt
    (set_state 2))
  
  ;; State 2: Complete
  (sequence
    (finalize)
    (set_state 0)))
```

## Code Propagation Summary
```
Child returns code
        │
        ▼
┌───────────────────────────────────────────────────────────────┐
│ Is it a PIPELINE code (9-12)?                                 │
│   YES → Composite handles internally, does NOT propagate      │
│   NO  → Continue checking                                     │
└───────────────────────────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────────────────────────┐
│ Is it a FUNCTION code (6-8)?                                  │
│   YES → S_Engine function boundary handles, translates to     │
│         application code, propagates to ChainTree             │
│   NO  → Continue checking                                     │
└───────────────────────────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────────────────────────┐
│ Is it an APPLICATION code (0-5)?                              │
│   YES → Pass through to ChainTree unchanged                   │
└───────────────────────────────────────────────────────────────┘
```

## Design Rationale

### Why Three Layers?

1. **Separation of concerns**: ChainTree doesn't need to know about S_Engine internals. Pipeline composites don't need to know about function boundaries.

2. **Composability**: New composite types can define their own interpretation of pipeline codes without affecting the rest of the system.

3. **Efficiency**: Pipeline codes are handled locally with no propagation overhead. Only codes that genuinely need to escape bubble up.

4. **Clarity**: When reading code, the return code tells you exactly what scope it affects:
   - `SE_CONTINUE` → tree level
   - `SE_FUNCTION_RESET` → function level
   - `SE_PIPELINE_TERMINATE` → composite level

### Relationship to ChainTree Events

ChainTree has its own event system (INIT, TICK, TERM, RESET, etc.). The application result codes map to ChainTree's expected return values:

| ChainTree Expectation | S_Engine Code |
|-----------------------|---------------|
| Continue normal execution | `SE_CONTINUE` |
| Pause and retain state | `SE_HALT` |
| Complete and cleanup | `SE_TERMINATE` |
| Restart from scratch | `SE_RESET` |
| Deactivate this branch | `SE_DISABLE` |

The S_Engine's function and pipeline codes are internal implementation details that ChainTree never sees.

## Quick Reference
```c
typedef enum {
    // APPLICATION RESULT CODES (pass through to ChainTree)
    SE_CONTINUE           = 0,   // Normal, continue
    SE_HALT               = 1,   // Pause, retain state
    SE_TERMINATE          = 2,   // Complete, cleanup
    SE_RESET              = 3,   // Restart
    SE_DISABLE            = 4,   // Deactivate
    SE_SKIP_CONTINUE      = 5,   // Skip siblings
    
    // FUNCTION RESULT CODES (handled at function boundary)
    SE_FUNCTION_HALT      = 6,   // Halt entire function
    SE_FUNCTION_RESET     = 7,   // Reset entire function
    SE_FUNCTION_TERMINATE = 8,   // Terminate entire function
    
    // PIPELINE RESULT CODES (handled by composite nodes)
    SE_PIPELINE_TERMINATE      = 9,   // End composite, success
    SE_PIPELINE_RESET_CONTINUE = 10,  // Reset children, keep going
    SE_PIPELINE_RESET_HALT     = 11,  // Reset children, pause
    SE_PIPELINE_DISABLE        = 12,  // Disable composite
} s_expr_result_t;
```
